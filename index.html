Original Paper

What Is AutoVLA?

The original paper, AutoVLA: A Vision-Language-Action Model for End-to-End Autonomous Driving with Adaptive Reasoning and Reinforcement Fine-Tuning, proposes a driving model that combines perception, reasoning, and planning in one generative model. Instead of predicting only text or high-level commands, AutoVLA predicts physical action tokens that can be decoded into a future driving trajectory.

The main idea is simple: a Vision-Language Model looks at camera videos, reads the vehicle state and driving command, and then outputs either a direct action or a short Chain-of-Thought reasoning followed by the final action. This makes the model flexible: it can be fast for simple scenes and more careful for complex scenes.

Paper Summary

Original AutoVLA In Simple Words

1. Vision-Language-Action

AutoVLA extends a VLM into a driving planner. The model receives camera frames and text prompts, then produces action tokens as language-like outputs.

2. Physical Action Tokens

Continuous trajectories are discretized into a codebook. Each token represents a physically meaningful short motion pattern for the ego car.

3. Adaptive Reasoning

The model can use fast thinking for easy cases or slow thinking with Chain-of-Thought for harder traffic scenes.

Method Overview

Step 1

Input: multi-view camera streams and ego state

AutoVLA takes front, front-left, and front-right camera videos. It also uses the vehicle's current velocity, acceleration, and a high-level driving instruction such as going straight or turning.

Step 2

Backbone: Qwen2.5-VL-3B

The paper uses a compact multimodal VLM backbone. This model can read image/video inputs and text prompts, then generate output tokens.

Step 3

Action tokenization

Continuous vehicle trajectories are converted into discrete physical action tokens. These tokens are added to the language model vocabulary, so trajectory prediction becomes a next-token prediction problem.

Step 4

Reasoning data

The paper builds reasoning supervision with a large VLM and also uses DriveLM-style annotations. The reasoning format describes the scene, important objects, likely intentions, and the final driving action.

Step 5

SFT: learn fast and slow thinking

In SFT, the model learns from both action-only examples and examples with Chain-of-Thought reasoning. This teaches the model two modes: direct action output and reasoning-before-action output.

Step 6

RFT / GRPO: improve planning and efficiency

After SFT, the paper applies reinforcement fine-tuning with GRPO and task rewards. The goal is to improve driving quality while penalizing unnecessary long reasoning in simple scenes.

Introduction

Problem And Contribution

Problem

Some VLM-based driving systems output text waypoints or high-level actions that may not be physically feasible. Also, many systems use one fixed reasoning style for every scene: always reason, or never reason.

AutoVLA's Solution

AutoVLA combines K=2048 action tokenization, SFT for dual-mode behavior, and RFT/GRPO to reduce unnecessary reasoning. For simple scenes it can directly output actions. For complex scenes it can use Chain-of-Thought first.

Framework

AutoVLA Framework In Detail

AutoVLA is built around one central idea: driving can be formulated as autoregressive token generation. The model receives visual observations and driving context, then generates either reasoning tokens, action tokens, or both. The action tokens are not ordinary words; they represent feasible vehicle motion patterns that can be decoded into a planning trajectory.

Framework diagram (from paper)

The official paper figure shows the full pipeline: multi-view video frames enter the Qwen2.5-VL encoder, the resulting embeddings are concatenated with the ego-state tokens, and the LM decoder then produces either a chain-of-thought prefix or goes directly to action tokens. The action tokens are looked up in the K=2048 codebook and decoded into a 3-second trajectory at 0.5-second intervals.

See autovla.github.io for the original figure.

Input

Multi-view temporal driving context

The paper uses three camera views: front, front-left, and front-right. Each view provides four sequential frames sampled at 2 Hz. The model also receives ego velocity, acceleration, historical state, and a high-level instruction such as turning or going straight.

Backbone

Qwen2.5-VL-3B as the multimodal decoder

The VLM converts the camera/video information and text prompt into a single sequence modeling problem. This makes reasoning text and driving actions part of the same output space.

Actions

Physical action tokens

Continuous future trajectories are discretized with a K-disk style action codebook. Tokens such as <action_0> to <action_2047> are added to the VLM vocabulary.

1. Action Tokenization

AutoVLA does not ask the language model to directly write floating-point waypoints. Instead, it converts short motion segments into discrete action tokens. This helps keep the output physically meaningful and makes planning compatible with next-token prediction.

2. Reasoning Data

The paper uses structured driving reasoning data. A reasoning example explains the scene, critical objects, intentions of surrounding agents, and why a final driving action is appropriate.

3. Unified Output Format

The same model can output text reasoning and action tokens in one sequence. This avoids using a separate reasoning model and a separate planner.

4. Trajectory Decoding

During inference, generated action tokens are mapped back through the codebook and rolled out into a continuous future trajectory for the ego car.

How the Model Is Prompted

AutoVLA uses a structured system prompt that defines the model's role, task, and expected reasoning format. The user message provides camera streams, ego state, and the driving instruction for each inference call. Figure S5 from the paper shows the exact prompt structure:

Figure S5 — image could not be loaded. View at: arxiv.org/html/2506.13757v3
Figure S5 (from paper): System Prompt and User Message of AutoVLA. The System Prompt (left) sets the model's role as an Advanced Driver Assistance System and specifies a five-step reasoning structure: (1) Scene Analysis, (2) Identification of Critical Objects, (3) Prediction of Critical Object Behavior, (4) Ego Vehicle Intent Reasoning, (5) Final Action Decision — selecting one lateral action (move forward / turn left / turn right / change lane) and one longitudinal action (stop / decelerate / maintain / accelerate). The User Message (right) carries the actual per-inference driving context: front, front-left, and front-right camera streams (4 sequential frames at 2 Hz each), current ego velocity and acceleration, and a high-level navigation instruction such as Keep Forward. This structure is what enables the adaptive fast/slow thinking behaviour — the system prompt instructs the model to use Chain-of-Thought only when it is necessary, and to produce a direct action otherwise.

Fast Thinking vs Slow Thinking

Fast Thinking

Fast thinking means the model directly predicts the final action tokens without producing a long Chain-of-Thought. This is useful for simple scenes: lane following, clear roads, or cases where the driving command is easy.

In the paper, this mode is important for runtime efficiency. If every scene used long reasoning, the model would be too slow for practical driving.

Slow Thinking

Slow thinking means the model first writes a structured reasoning process, then outputs the final action tokens. This is useful when the scene has pedestrians, intersections, traffic lights, or other agents with uncertain intentions.

The paper's goal is not to always reason. The goal is adaptive reasoning: reason when it helps, skip it when the scene is straightforward.

Training Strategy: SFT Then RFT

Supervised Fine-Tuning (SFT)

SFT teaches the model the basic output behavior. Some training examples contain only final action tokens, while others contain Chain-of-Thought followed by final action tokens. This teaches the model both fast and slow modes inside one architecture.

The SFT objective includes normal language-model token loss and extra focus on action-token prediction, because planning quality depends strongly on choosing the correct physical action tokens.

Reinforcement Fine-Tuning (RFT / GRPO)

After SFT, the paper applies GRPO-based reinforcement fine-tuning. The model samples candidate outputs, receives task rewards, and updates the policy using group-relative advantages.

RFT has two purposes: improve trajectory quality and reduce unnecessary reasoning. A CoT length penalty encourages the model to avoid slow thinking when the scene does not need it.

My Implementation

What I Actually Did

What ran

  • Set up Qwen2.5-VL-3B with the AutoVLA codebase and got it running on a V100 cluster.
  • Preprocessed the full nuScenes v1.0 trainval split into per-sample JSON records — 19,030 train and 5,569 val scenes.
  • Integrated DriveLM annotations as Chain-of-Thought supervision fields for the SFT stage.
  • Ran SFT and RFT trials via SLURM. The SFT took about 30 hours across multiple runs. RFT ran for another ~49 hours per trial.
  • Discovered through evaluation logs that the checkpoint never produced direct action tokens — it always fell back to logits. Spent time auditing why (see Improvement section) and built a two-phase repair that fixed this.
  • Generated Figure S6-style qualitative examples comparing fast and slow thinking modes.

What I skipped

  • Multi-dataset training (nuPlan + Waymo + CARLA) — storage and compute made this impractical for this project.
  • The paper's official checkpoints are marked "Coming Soon" — there's nothing to compare against directly.
  • CARLA closed-loop simulation.
  • nuPlan NAVSIM PDMS evaluation — requires the full nuPlan infrastructure.
The gap between my numbers and the paper is real and expected. Most of it comes from using a single dataset vs four, and from the action-token fallback issue that the repair partially fixes.

Proposed Improvement

Two-Phase Action-Token Repair

Through systematic debugging we discovered that our trained SFT checkpoint never autoregressively emits <action_*> tokens — it always outputs natural language ("turn_right", "go_straight") and falls back to logit-based action selection. This is the primary cause of the large L2 gap.

Root Cause Diagnosis

What IS working: tokenizer mapping (verified: <action_0>→151665), training labels (10 action tokens per sample confirmed), gradient flow (loss 8.07→4.31 in one backward step), text-only action generation (text_action_rate reaches 1.0 in 1 epoch with short prompts).
What fails: in the multimodal setting with the long CoT prompt, the model's natural-language prior overwhelms the action-token loss signal. The model outputs "turn_right" or "go_straight" as text instead of <action_i> tokens.

The Repair Method

Phase 1

Text-only warm-up (1 epoch)

Short text-only prompt, no video, system message: "You output only AutoVLA action tokens. No words." Loss restricted to the 10 action-token positions. Gate loss pushes probability mass from natural-language tokens into the action group.

Result: text_action_rate jumps from 0.000 → 1.000 in 1 epoch.

Phase 2

Multimodal repair (20 steps)

Uses the exact same prompt as evaluation (get_prompt() → "<answer>\nThe final output action is: "). Loss computed only on the 10 action-token positions. Generation is checked every 20 steps on held-out samples. Training stops at step 20 to avoid mode collapse.

Result: multimodal text_action_rate 0.000 → 1.000.

Remaining Gap Analysis

After the repair, L2@1s is 5.55 m vs the paper's 0.22 m. The remaining 25× gap is not from the action-token decoding mechanism (now fixed), but from the quality of the underlying SFT checkpoint:

  • Training data: paper uses multi-dataset training (nuPlan + nuScenes + Waymo); we use nuScenes only.
  • No RFT: paper applies GRPO reinforcement fine-tuning after SFT; we do not.
  • Mode collapse: repair trained on 20 samples; model converges to a narrow set of action sequences.
  • Official weights not released: paper's checkpoints are marked "Coming Soon"; we trained from scratch.

Observations

Gap Analysis

What We Can Say Now

  • nuScenes preprocessing and validation scale look reasonable.
  • The evaluation can finish, and the NaN problem is no longer the main blocker.
  • The qualitative figure can be generated and used for presentation.
  • The current L2 values are not paper-level reproduction results.

Likely Sources Of The Gap

  • The model often uses logits fallback instead of clean action-token generation.
  • This repo uses nuScenes-only training, while the paper uses mixed-dataset training.
  • Checkpoint choice and SFT/RFT settings may differ from the paper.
  • Table S2 metric indexing and collision protocol may not be fully aligned yet.

Summary

What This Project Found

Root Cause Identified

  • The SFT checkpoint uses 100% logit fallback — never generates <action_*> tokens directly.
  • The model's natural-language prior (from CoT training) overwhelms the action-token signal in the multimodal setting.
  • The tokenizer, labels, and gradient flow are all verified correct — the problem is behavioral, not architectural.

Fix Verified

  • Two-phase action-token repair: model now generates action tokens directly, no fallback needed.
  • L2@1s improves 11.7% (6.29 → 5.55 m) on 200 validation samples.
  • Remaining gap to paper (0.22 m) is from checkpoint quality, not decoding.
